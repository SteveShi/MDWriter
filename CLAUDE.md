## Build & Verify

The Xcode project is generated by XcodeGen — `project.yml` is the source of truth. Regenerate after any structural, settings, dependency, or version change.

```bash
xcodegen generate --spec project.yml
xcodebuild -project MDWriter.xcodeproj -scheme MDWriter -configuration Debug \
  -destination 'platform=macOS' \
  CODE_SIGNING_ALLOWED=NO CODE_SIGNING_REQUIRED=NO build
```

Run a compile check before reporting code changes complete. Do not hand-edit the generated `MDWriter.xcodeproj` internals.

## Architecture (big picture)

Native SwiftUI + SwiftData macOS Markdown editor. Minimum macOS 14.0, Swift 6.0+. App lifecycle starts at `MDWriter/App/MDWriterApp.swift`; the document experience is composed through `LibraryView` + `ContentView` + editor feature views + shared menu commands. SwiftData persistence is centralized through the app-level `ModelContainer`.

Layout under `MDWriter/`:

- `App/` — entry point, theme, root content, entitlements, Info.plist
- `Core/Models/` — SwiftData models (keep schema churn low)
- `Core/Services/` — shared services (e.g. image handling)
- `Core/Integrations/` — `Updater.swift` (Sparkle) and `WhatsNewConfiguration.swift`
- `Features/<FeatureName>/` — `AI/`, `Backup/`, `Editor/`, `Export/`, `Library/`, `Settings/`
- `Resources/Localizable.xcstrings` — all user-facing strings
- `UIComponents/` — shared SwiftUI/AppKit wrappers and commands

Editor behavior should flow through existing editor controllers and settings before adding new state. Custom regex-based syntax highlighting is "MarkX" (no webview). Apple Foundation Models power the on-device AI assistant (requires macOS 26 + Apple Silicon at runtime).

## Project conventions that bite

- **Localization is mandatory.** Every visible label/button/menu/tooltip/placeholder/error must use `LocalizedStringKey`, `String(localized:)`, or the surrounding pattern, with keys in `MDWriter/Resources/Localizable.xcstrings`. No hard-coded UI strings.
- **No legacy/compat shims** unless explicitly requested. Don't multiply entities; prefer extending existing types over creating parallel ones.
- **Main-actor for UI work.** Avoid unsafe cross-actor mutation.
- **Bundle ID convention:** `com.steveshi.appname` for new app targets/bundle products.
- **Clean up** any temporary `*.log` / `*.txt` files before finishing.

## Release workflow — do not "simplify"

`.github/workflows/release.yml` contains a known-good Sparkle 2.x path. Treat these as load-bearing:

- The `Extract Version from Changelog` and `Extract Release Notes (en & zh)` steps.
- Private-key cleanup via `tr -dc 'A-Za-z0-9+/='`.
- `DYLD_FRAMEWORK_PATH` set to the Sparkle tools directory.
- Stdin signing: `echo "$KEY" | generate_appcast --ed-key-file -`.

Never modify version/changelog extraction logic, and don't refactor the release workflow if the task is unrelated to releases.

## Versioning

`project.yml` is the version source of truth. When bumping:

1. Update `MARKETING_VERSION` and `CURRENT_PROJECT_VERSION` in `project.yml`.
2. Update `CHANGELOG.md`, keeping the bilingual shape Sparkle expects: **English first, separator line, then Chinese**.
3. Regenerate the Xcode project.
