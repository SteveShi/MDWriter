# MDWriter

Native Markdown editor for macOS, built with SwiftUI, SwiftData, XcodeGen, Sparkle, and localized English/Chinese release notes.

## Quick Reference

- Generate project: `xcodegen generate --spec project.yml`
- Build: `xcodebuild -project MDWriter.xcodeproj -scheme MDWriter -configuration Debug -destination 'platform=macOS' CODE_SIGNING_ALLOWED=NO CODE_SIGNING_REQUIRED=NO build`
- Release workflow: `.github/workflows/release.yml`
- Version source: `project.yml`
- Release notes: `CHANGELOG.md`
- Localization: `MDWriter/Resources/Localizable.xcstrings`
- Minimum macOS: 14.0
- Swift: 6.0 or higher

## Workflow Rules

- Communicate with the user in Chinese.
- Read this file before making non-trivial changes in this repository.
- Keep changes minimal and directly related to the task. Entities should not be multiplied without necessity.
- Do not introduce legacy fallback paths or compatibility shims unless the user explicitly asks for them.
- Clean up temporarily generated `*.log` and `*.txt` files before finishing the task.
- Prefer existing project APIs, services, and models. Verify relevant API behavior before adding a new interface.
- When modifying code, run a compile check before reporting completion.
- Xcode projects are generated by XcodeGen. Edit `project.yml`, then regenerate the project with XcodeGen when project structure, settings, dependencies, or version metadata change.
- Do not hand-edit generated Xcode project structure unless there is a clear emergency and the user has approved that direction.

## Release Workflow Guardrails

- Never modify the version extraction or changelog extraction logic in `.github/workflows/release.yml`.
- In particular, do not change the sections named for extracting the version or release notes, including `Extract Version from Changelog` and `Extract Release Notes (en & zh)`.
- The current Sparkle 2.x GitHub Actions release path is known-good:
  - Clean the private key with `tr -dc 'A-Za-z0-9+/='`.
  - Set `DYLD_FRAMEWORK_PATH` to the Sparkle tools directory.
  - Sign through stdin with `echo "$KEY" | generate_appcast --ed-key-file -`.
- Do not "simplify" the release workflow if the requested task is unrelated to releases.
- When updating the app version, update both `project.yml` and `CHANGELOG.md`.
- Changelog entries must keep Sparkle-friendly bilingual release notes: English first, a separator line, then Chinese.

## Key Patterns

- App lifecycle starts at `MDWriter/App/MDWriterApp.swift`.
- Main document experience is composed through `LibraryView`, `ContentView`, editor feature views, and shared menu commands.
- SwiftData persistence is centralized through the app-level `ModelContainer`.
- Editor behavior should flow through existing editor controllers and settings before adding new state.
- Sparkle update checks are handled by `Core/Integrations/Updater.swift`.
- WhatsNew content is configured through `Core/Integrations/WhatsNewConfiguration.swift`.
- Keep feature code under `MDWriter/Features/<FeatureName>/` and shared primitives under `Core/` or `UIComponents/`.

## Coding Conventions

### Swift

- Swift version must be 6.0 or higher.
- Use SwiftUI-native patterns and SwiftData-native persistence.
- Prefer small, focused views and services over broad manager objects.
- Use `// MARK: -` to organize substantial files.
- Keep access control conservative. Use `private` when it clarifies ownership or prevents accidental coupling.
- UI-affecting work belongs on the main actor. Avoid unsafe cross-actor state mutation.
- Do not add hard-coded user-facing string literals. Localizable content must use `Localizable.xcstrings`.
- Do not create duplicated models, services, or wrappers when an existing project type can be extended cleanly.
- Bundle identifiers must follow `com.steveshi.appname` for new app targets or bundle-style products.

### Localization

- Every user-visible label, button title, menu item, tooltip, placeholder, error, and release-facing text must be localizable.
- Use `LocalizedStringKey`, `String(localized:)`, or the existing localization approach already used around the call site.
- Add or update keys in `MDWriter/Resources/Localizable.xcstrings` when introducing new visible text.
- Keep Chinese and English wording natural; do not leave raw development keys visible in the UI.

### SwiftData

- Keep model definitions in `MDWriter/Core/Models/`.
- Avoid schema churn unless the task requires it.
- When changing persistence behavior, verify save/load paths and migration impact.
- Do not silently discard user content. Prefer explicit errors or recoverable flows.

### Shell And Automation

- Prefer `zsh` for project scripts.
- Use `set -euo pipefail` in new shell scripts.
- Scripts should resolve paths relative to themselves or the repository root, not the caller's current directory.
- Do not leave temporary build, note, or diagnostic files behind.

## Build And Verification

- For code changes, run:

```bash
xcodegen generate --spec project.yml
xcodebuild -project MDWriter.xcodeproj -scheme MDWriter -configuration Debug -destination 'platform=macOS' CODE_SIGNING_ALLOWED=NO CODE_SIGNING_REQUIRED=NO build
```

- If only documentation changed, a compile check is not required, but still verify the working tree and the edited file.
- If a build cannot be run, explain exactly why and include the highest-signal verification that was completed.

## Version And Changelog

- `project.yml` is the version source of truth for generated Xcode settings.
- When updating versions, update:
  - `MARKETING_VERSION`
  - `CURRENT_PROJECT_VERSION`
  - `CHANGELOG.md`
- Keep `CHANGELOG.md` bilingual:
  - English release notes first.
  - A separator line between languages.
  - Chinese release notes second.
- Preserve the release-note shape expected by Sparkle 2.9.0 and the existing GitHub Actions workflow.

## Design System

- Audience: writers who want a focused, native, trustworthy Markdown editor.
- Feel: quiet professional writing desk; native macOS first, not a web app imitation.
- Typography: readability first, with careful English and CJK spacing.
- Layout: efficient three-column/library/editor flows, restrained panels, no decorative clutter.
- Controls: prefer standard macOS and SwiftUI controls unless a custom writing interaction is clearly better.
- Themes: keep themes coherent across editor, sidebars, export previews, settings, and menus.
- Accessibility: preserve keyboard navigation, menu command discoverability, dynamic type friendliness where practical, and sufficient contrast.

## Do Not Touch Without Explicit Reason

- `.github/workflows/release.yml` version extraction logic.
- `.github/workflows/release.yml` release-note extraction logic.
- Known-good Sparkle key cleanup, `DYLD_FRAMEWORK_PATH`, and stdin signing steps.
- Generated Xcode project internals when `project.yml` is the proper source of truth.
- Existing localized strings by replacing them with hard-coded text.
